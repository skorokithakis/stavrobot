# Decision log

Architectural decisions made during discussion. Only decisions from user discussion are recorded here. Format: "Decision: Reason".

- Internal HTTP listener on port 3001 for inter-service communication: Avoids distributing the app password to every container that needs to call back. Network-level isolation instead of auth.
- Init scripts declared in manifest, not conventional filenames: Needed a place to put the `async` flag. Manifest field is explicit and extensible.
- Init scripts run on both install and update: Ensures setup steps are re-run when plugin code changes (e.g., new dependencies, schema migrations).
- Init script output is returned to the agent: Lets init scripts report generated data (credentials, setup results) back to the agent.
- Async execution is opt-in per tool/init via manifest: Sync is the default. Async adds complexity and should only be used for long-running operations.
- Async timeout is 5 minutes, sync timeout is 30 seconds: Async scripts don't block the runner, so a longer timeout is safe.
- Unified tool-runner and plugin-runner into a single plugin-runner container: Eliminates duplicate code and simplifies the architecture. Unix user isolation per plugin is sufficient — there is no need for container-level separation between locally created tools and git-installed plugins. Editability is determined by the presence of a `.git` directory and enforced by the main app before dispatching coding tasks.
- Agent/subagent model: agents are rows in the `agents` table with their own conversation history, system prompt, and tool whitelist. Interlocutors are contact records assigned to agents for inbound message routing — they no longer carry per-interlocutor instructions. Agent-to-agent messaging is done via the queue with `source: "agent"`. Tool whitelisting uses `["*"]` for all tools, `[]` for none, or an explicit list. Identity soft-deletion (setting `identifier` to null) allows removing a channel identity without losing the interlocutor record or its message history.
- Agent references always include both name and ID: when an agent appears in the LLM context (e.g., as a message sender), it is shown as "name (ID: N)". Agents are addressed by numeric ID in tool calls, since names are not unique and serve more as descriptions.
- Structured tool payloads are TOON-encoded before entering agent context: Keeps the context format consistent and avoids leaking raw JSON transport shapes into reasoning.
- Plugin permissions are set via the web UI only, not by the LLM: allowing the LLM to modify its own tool restrictions would defeat the purpose of the permission system.
- Audio handled as regular file attachments, STT moved to a plugin: removes the dedicated `audio`/`audioContentType` fields from the API and queue in favor of the unified `files`/`attachments` path. All bridges send audio through the same file attachment mechanism as documents and images. Transcription is an explicit plugin tool call rather than implicit core behavior. Cleaner architecture, one path for all files, no special cases.
- Separate `allowed_tools` and `allowed_plugins` fields for subagent access control: `allowed_tools` is validated against a hard allowlist of safe core tools (no `execute_sql`, `manage_agents`, `manage_plugins`, etc.). Plugin access is controlled by `allowed_plugins` with granular per-plugin and per-tool scoping via dot notation. `run_plugin_tool` is implicitly available when `allowed_plugins` is non-empty. This prevents privilege escalation while allowing fine-grained plugin access.
